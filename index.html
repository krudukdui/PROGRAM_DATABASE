Database System — Concepts, Types & DBMS

1. ความหมายและองค์ประกอบ

💡

ฐานข้อมูลคืออะไร?

ฐานข้อมูล (Database) คือ การรวบรวมข้อมูลที่มีความสัมพันธ์กัน จัดเก็บอย่างเป็นระบบในคอมพิวเตอร์ เพื่อให้สามารถค้นหา เรียกใช้ และจัดการข้อมูลได้อย่างมีประสิทธิภาพ และลดความซ้ำซ้อนของข้อมูล

🌍 ตัวอย่างในชีวิตจริง
🏦ธนาคาร — ข้อมูลลูกค้า, บัญชี, ธุรกรรม
🏫โรงเรียน — ข้อมูลนักเรียน, ครู, วิชาเรียน
🏥โรงพยาบาล — ข้อมูลผู้ป่วย, แพทย์, ยา
🛒ร้านค้าออนไลน์ — สินค้า, คำสั่งซื้อ, ลูกค้า

🏗️ องค์ประกอบของระบบฐานข้อมูล

🔑 คำศัพท์สำคัญที่ต้องรู้

Table (ตาราง)
การจัดเก็บข้อมูลในรูปแบบแถวและคอลัมน์ เช่น ตารางข้อมูลนักเรียน
Field (ฟิลด์)
คอลัมน์ในตาราง แต่ละ Field เก็บข้อมูลประเภทเดียวกัน เช่น ชื่อ, อายุ
Record (ระเบียน)
แถวหนึ่งในตาราง แทนข้อมูลของสิ่งหนึ่งๆ เช่น ข้อมูลนักเรียน 1 คน
Primary Key (PK)
ฟิลด์ที่ระบุ Record ได้โดยไม่ซ้ำกัน และต้องไม่ว่างเปล่า เช่น รหัสนักเรียน
Foreign Key (FK)
ฟิลด์ที่อ้างอิง Primary Key ของตารางอื่น ใช้เชื่อมความสัมพันธ์ระหว่างตาราง
DBMS
Database Management System — ซอฟต์แวร์ที่ใช้จัดการฐานข้อมูล เช่น Microsoft Access
Query
การสอบถามหรือดึงข้อมูลจากฐานข้อมูล โดยระบุเงื่อนไขที่ต้องการ
Schema
โครงสร้างหรือแผนผังของฐานข้อมูล บอกว่ามีตาราง Field และความสัมพันธ์อะไรบ้าง

2. ชนิดของฐานข้อมูล

🌳 1. ฐานข้อมูลแบบลำดับชั้น (Hierarchical Database)

🌲

จัดเก็บข้อมูลในรูปแบบ ต้นไม้ (Tree Structure) มีความสัมพันธ์แบบ Parent–Child ข้อมูล Parent หนึ่งตัวมี Child ได้หลายตัว แต่ Child หนึ่งตัวมี Parent ได้เพียงตัวเดียว

✅ ค้นหาข้อมูลตามลำดับชั้นได้รวดเร็ว
❌ ยืดหยุ่นน้อย แก้โครงสร้างยาก
📌 ตัวอย่าง: IBM IMS, Windows Registry

🕸️ 2. ฐานข้อมูลแบบเครือข่าย (Network Database)

🕸️

พัฒนาจาก Hierarchical โดย Child หนึ่งตัวสามารถมี Parent ได้หลายตัว ทำให้โครงสร้างยืดหยุ่นกว่า แต่ซับซ้อนกว่า ความสัมพันธ์เป็น M:N

📦 สินค้า A
📦 สินค้า B
🏭 ผู้ผลิต 1
🏭 ผู้ผลิต 2
⟵ สินค้าหนึ่งชิ้นอาจมาจากหลายผู้ผลิต ⟶
✅ รองรับความสัมพันธ์ M:N ได้
❌ โครงสร้างซับซ้อน ยากต่อการดูแล
📌 ตัวอย่าง: IDMS, IDS

🗃️ 3. ฐานข้อมูลเชิงสัมพันธ์ (Relational Database) ⭐

🗃️

จัดเก็บข้อมูลในรูปแบบตาราง (Table/Relation) เชื่อมตารางด้วย Primary Key และ Foreign Key เป็นรูปแบบที่นิยมใช้มากที่สุดในปัจจุบัน ใช้ภาษา SQL ในการจัดการข้อมูล

🔷 4. ฐานข้อมูลเชิงวัตถุ (Object-Oriented Database)

3. คุณสมบัติของฐานข้อมูลที่ดี

⭐ ลักษณะและคุณสมบัติของฐานข้อมูลที่ดี

ฐานข้อมูลที่ออกแบบมาดี ควรมีคุณสมบัติสำคัญ 7 ประการ ดังนี้

01
🔄
ความซ้ำซ้อนน้อย
(Minimal Redundancy)
จัดเก็บข้อมูลแต่ละชุดในที่เดียว ไม่เก็บข้อมูลเดิมซ้ำหลายที่
❌ เก็บชื่อลูกค้าไว้ทั้งตาราง Orders และ Invoice
✅ เก็บชื่อลูกค้าในตาราง Customers ตารางเดียว
02
✅
ความถูกต้อง
(Data Integrity)
มีกลไกป้องกันข้อมูลผิดพลาด เช่น กำหนดชนิดข้อมูลและ Constraint
✅ กำหนด Data Type, Required Field, Validation Rule
03
🔗
ความสม่ำเสมอ
(Consistency)
ข้อมูลในทุกส่วนต้องสอดคล้องกัน ไม่ขัดแย้งกัน
✅ ใช้ Transaction เพื่อให้การเปลี่ยนแปลงสมบูรณ์ทุกครั้ง
04
🤝
การแชร์ข้อมูล
(Data Sharing)
ผู้ใช้หลายคนสามารถเข้าถึงข้อมูลพร้อมกันได้
✅ พนักงานหลายคนดูข้อมูลลูกค้าได้พร้อมกัน
05
🔒
ความปลอดภัย
(Security)
มีระบบกำหนดสิทธิ์การเข้าถึงข้อมูล ป้องกันผู้ไม่มีสิทธิ์
✅ กำหนด Username/Password และระดับสิทธิ์
06
🔍
ค้นหาง่าย
(Easy Retrieval)
สามารถค้นหาและดึงข้อมูลที่ต้องการได้รวดเร็ว
✅ ใช้ Index ในฟิลด์ที่ค้นหาบ่อย ทำให้ Query เร็วขึ้น
07
🔙
สำรองและกู้คืน
(Backup & Recovery)
มีระบบสำรองข้อมูลสม่ำเสมอ และสามารถกู้คืนได้เมื่อเกิดความผิดพลาด
✅ กำหนด Backup อัตโนมัติทุกวัน และทดสอบการ Restore

5. ข้อดีและข้อเสียของการใช้ฐานข้อมูล

✅ ข้อดีของการใช้ฐานข้อมูล

📉
ลดความซ้ำซ้อนของข้อมูล (Reduced Data Redundancy)
ข้อมูลแต่ละชุดเก็บในที่เดียว ไม่มีการเก็บข้อมูลซ้ำๆ ในหลายไฟล์ ประหยัดพื้นที่จัดเก็บและลดความสับสน
🔗
ความสม่ำเสมอของข้อมูล (Data Consistency)
เมื่อแก้ไขข้อมูลในที่เดียว การเปลี่ยนแปลงจะสะท้อนทั่วทั้งระบบ ไม่มีข้อมูลขัดแย้งกัน
🤝
แชร์ข้อมูลได้ (Data Sharing)
ผู้ใช้หลายคนในองค์กรสามารถเข้าถึงและใช้ข้อมูลร่วมกันได้พร้อมกัน
🔒
ความปลอดภัยของข้อมูล (Data Security)
สามารถกำหนดสิทธิ์การเข้าถึงข้อมูลได้ละเอียด ป้องกันการเข้าถึงโดยไม่ได้รับอนุญาต
🛡️
ความถูกต้องของข้อมูล (Data Integrity)
มีกลไกตรวจสอบความถูกต้องของข้อมูลอัตโนมัติ ด้วย Constraint และ Validation Rule
💾

1. หลักการออกแบบฐานข้อมูลเบื้องต้น

💡

การออกแบบฐานข้อมูลที่ดีคืออะไร?

การออกแบบฐานข้อมูล คือการวางแผนโครงสร้างการจัดเก็บข้อมูลให้ ถูกต้อง ครบถ้วน ไม่ซ้ำซ้อน และสามารถนำข้อมูลกลับมาใช้งานได้อย่างมีประสิทธิภาพ

🎯 หลักการสำคัญในการออกแบบ

🚫
ลดความซ้ำซ้อน (Reduce Redundancy)
ไม่เก็บข้อมูลเดิมซ้ำหลายที่ เช่น ชื่อแผนกควรอยู่ในตารางแผนกเพียงที่เดียว
✅
ความถูกต้องของข้อมูล (Data Integrity)
ข้อมูลต้องสอดคล้องกัน เปลี่ยนที่เดียวแล้วถูกต้องทุกที่
🔍
ค้นหาได้ง่าย (Easy to Query)
โครงสร้างตารางต้องทำให้สามารถดึงข้อมูลได้รวดเร็วและยืดหยุ่น
📈
รองรับการขยาย (Scalability)
ออกแบบให้เพิ่มข้อมูลหรือ Field ใหม่ในอนาคตได้โดยไม่กระทบโครงสร้างเดิม
🔒
ความปลอดภัย (Security)
กำหนดสิทธิ์การเข้าถึงข้อมูลให้เหมาะสมกับผู้ใช้แต่ละระดับ
⚡
ประสิทธิภาพ (Performance)
โครงสร้างที่ดีช่วยให้การบันทึกและเรียกใช้ข้อมูลทำได้รวดเร็ว

2. การวิเคราะห์ความต้องการของระบบ

🔎 การวิเคราะห์ความต้องการของระบบ (System Requirements Analysis)

📋

ทำไมต้องวิเคราะห์ความต้องการก่อน?

ก่อนสร้างฐานข้อมูล ต้องเข้าใจว่าระบบต้องการเก็บข้อมูลอะไร ใครเป็นผู้ใช้งาน และต้องการรายงานอะไรบ้าง เพื่อให้การออกแบบตรงกับความต้องการจริง

🛠️ เทคนิคการวิเคราะห์ความต้องการ

🗣️
สัมภาษณ์ (Interview)
พูดคุยกับผู้ใช้งานโดยตรง เพื่อเข้าใจกระบวนการทำงานและความต้องการ
📄
ศึกษาเอกสาร (Document Study)
วิเคราะห์แบบฟอร์ม รายงาน และเอกสารที่ใช้งานอยู่ในปัจจุบัน
👁️
สังเกตการณ์ (Observation)
สังเกตการทำงานจริงของผู้ใช้เพื่อหาจุดที่ต้องปรับปรุง
📊
แบบสอบถาม (Questionnaire)
รวบรวมข้อมูลจากผู้ใช้จำนวนมากพร้อมกันอย่างมีระบบ

📌 ตัวอย่าง: วิเคราะห์ระบบทะเบียนนักเรียน

❓ ต้องเก็บข้อมูลอะไร?
ข้อมูลนักเรียน, วิชาเรียน, ครูผู้สอน, เกรด, ห้องเรียน
❓ ใครใช้งานระบบบ้าง?
ครู, นักเรียน, เจ้าหน้าที่ทะเบียน, ผู้อำนวยการ
❓ ต้องการรายงานอะไร?
รายชื่อนักเรียน, ผลการเรียน, สรุปเกรด, รายวิชาที่เปิดสอน
❓ Entity ที่ได้จากการวิเคราะห์?
นักเรียน วิชา ครู การลงทะเบียน ห้องเรียน

4. Normalization (1NF, 2NF, 3NF)

🔢 Normalization — การทำให้เป็นบรรทัดฐาน

กระบวนการปรับโครงสร้างตารางให้ถูกต้อง ลดความซ้ำซ้อนของข้อมูล

1NF
First Normal Form
กฎ: ทุก Field ต้องมีค่าเดียว (Atomic Value) และมี Primary Key
❌ วิชา: "คณิต, วิทย์, อังกฤษ" (หลายค่าในช่องเดียว)
✅ แยกเป็นหลาย Record แต่ละ Record มีวิชาเดียว
2NF
Second Normal Form
กฎ: ต้องอยู่ใน 1NF แล้ว และทุก Non-Key Field ต้องขึ้นกับ Primary Key ทั้งหมด (ไม่มี Partial Dependency)
❌ ชื่อวิชาขึ้นกับแค่ รหัสวิชา (ไม่ใช่ทั้ง PK)
✅ แยกตารางวิชาออกมาต่างหาก
3NF
Third Normal Form
กฎ: ต้องอยู่ใน 2NF แล้ว และไม่มี Transitive Dependency (Field ไม่ขึ้นกับ Field อื่นที่ไม่ใช่ PK)
❌ รหัสนักเรียน → รหัสแผนก → ชื่อแผนก (ห้ามมี)
✅ แยกตารางแผนกออกมา เก็บแค่ รหัสแผนก ในตารางนักเรียน

5. การออกแบบตารางและกำหนด Primary Key

🗂️ การออกแบบตารางที่ดี

🔑

Primary Key (PK) คืออะไร?

Primary Key คือ Field หรือกลุ่ม Field ที่ ระบุ Record ได้โดยไม่ซ้ำกัน ในตาราง มีกฎสำคัญ 2 ข้อ คือ ห้ามซ้ำกัน (Unique) และห้ามว่าง (NOT NULL)

📏 หลักการเลือก Primary Key

✅
ค่าไม่ซ้ำกันเลย (Unique)

ต้องระบุแต่ละ Record ได้เพียง Record เดียว เช่น รหัสนักเรียน ST001 มีแค่คนเดียวเสมอ

✅
ห้ามว่างเปล่า (NOT NULL)

ทุก Record ต้องมีค่า PK เสมอ จะเว้นว่างไม่ได้

✅
ค่าไม่เปลี่ยนแปลง (Stable)

ควรเป็น Field ที่ไม่ค่อยเปลี่ยน เช่น รหัส ไม่ควรใช้ชื่อ เพราะชื่ออาจซ้ำหรือเปลี่ยนได้

❌
ห้ามใช้ข้อมูลที่อาจซ้ำ

เช่น ชื่อ, เบอร์โทร, วันเกิด — ล้วนอาจซ้ำกันได้ ไม่เหมาะเป็น PK

Datasheet View — สร้างตารางแบบง่าย คล้าย Excel คลิกเพิ่ม Column ได้เลย เหมาะสำหรับผู้เริ่มต้น

1. ความหมายของความสัมพันธ์ระหว่างตาราง

🔗

ความสัมพันธ์ระหว่างตาราง (Table Relationship) คืออะไร?

ความสัมพันธ์ระหว่างตาราง คือ การเชื่อมโยงข้อมูลจากตารางหนึ่งไปยังอีกตาราง โดยใช้ Field ที่มีค่าเดียวกันเป็นตัวเชื่อม ทำให้สามารถดึงข้อมูลที่กระจายอยู่ในหลายตารางมาแสดงร่วมกันได้อย่างถูกต้องและมีประสิทธิภาพ

🌍 ทำไมต้องมีความสัมพันธ์ระหว่างตาราง?

🚫
ลดความซ้ำซ้อน
แทนที่จะเก็บชื่อแผนกซ้ำในทุก Record ของนักเรียน ให้เก็บแค่รหัสแผนก แล้วดึงชื่อแผนกจากตารางแผนกเมื่อต้องการ
✅
รักษาความถูกต้อง
เมื่อแก้ไขข้อมูลที่ตารางหลัก เช่น เปลี่ยนชื่อแผนก ข้อมูลทุกที่จะอัปเดตพร้อมกันทันที ไม่ต้องแก้ทีละ Record
🔍
ดึงข้อมูลข้ามตาราง
สามารถสร้าง Query เพื่อดึงข้อมูลจากหลายตารางพร้อมกัน เช่น แสดงชื่อนักเรียนพร้อมชื่อแผนกในผลลัพธ์เดียว
🏗️
โครงสร้างข้อมูลที่ดี
การออกแบบฐานข้อมูลที่ดีตามหลัก Normalization จะแบ่งข้อมูลออกเป็นหลายตารางที่มีความสัมพันธ์กัน

🔑 Primary Key vs Foreign Key — กุญแจสำคัญของความสัมพันธ์

PK

Primary Key (คีย์หลัก)

✅ ค่าไม่ซ้ำ (Unique) ✅ ห้ามว่าง (NOT NULL) ✅ ค่าไม่เปลี่ยนแปลง (Stable)
เช่น: DeptID ในตาราง tblDepts
🔗
เชื่อมกัน

5. Referential Integrity (ความถูกต้องของการอ้างอิง)

🛡️ Referential Integrity คืออะไร?

🛡️

Referential Integrity (RI)

Referential Integrity คือ กฎที่ Access บังคับใช้เพื่อรักษาความสัมพันธ์ระหว่างตารางให้ถูกต้องสม่ำเสมอ กำหนดว่า Foreign Key ในตารางลูกต้องมีค่าที่ตรงกับ Primary Key ในตารางหลักเสมอ จะมีค่า FK ที่ "ลอยอยู่" โดยไม่มี PK รองรับไม่ได้

📜 กฎของ Referential Integrity

กฎ 1
❌ ไม่สามารถเพิ่ม Record ใน Child Table ที่ไม่มีข้อมูลใน Parent Table
❌ เพิ่มนักเรียนที่มี DeptID = "XYZ" — แต่ไม่มีแผนก XYZ ในตาราง tblDepts → Access จะแสดง Error
✅ เพิ่มนักเรียนที่มี DeptID = "IT" — ต้องมีแผนก IT ในตาราง tblDepts ก่อน → บันทึกได้
กฎ 2
❌ ไม่สามารถลบ Record ใน Parent Table ถ้ายังมี Record ใน Child Table อ้างอิงอยู่
❌ ลบแผนก IT — แต่ยังมีนักเรียนที่มี DeptID = "IT" อยู่ → Access จะแสดง Error
✅ ลบนักเรียนทุกคนในแผนก IT ก่อน แล้วจึงลบแผนก IT → ลบได้
กฎ 3
❌ ไม่สามารถเปลี่ยน PK ใน Parent Table ถ้ายังมี FK ใน Child Table อ้างอิงอยู่ (ถ้าไม่เปิด Cascade Update)
❌ เปลี่ยน DeptID จาก "IT" → "TECH" — แต่มีนักเรียนอ้างอิง "IT" อยู่ → Error (ถ้าไม่เปิด Cascade Update)
✅ เปิด Cascade Update → Access อัปเดต DeptID ทุกที่อัตโนมัติ → ทำได้

⚡ RI เปิด vs ปิด — ผลที่แตกต่าง

🔓 ปิด Referential Integrity
❌ อาจมี FK ที่ไม่มี PK รองรับ ("Orphan Records")
❌ ลบ Parent ได้แม้มี Child อ้างอิงอยู่
❌ ข้อมูลไม่สม่ำเสมอ อาจผิดพลาดโดยไม่รู้ตัว
✅ ยืดหยุ่นกว่า เหมาะกับข้อมูลเก่าที่ไม่สมบูรณ์
🔒 เปิด Referential Integrity
✅ FK ทุกค่าต้องมี PK รองรับเสมอ
✅ ป้องกันการลบ Parent ที่ยังมี Child
✅ ข้อมูลสม่ำเสมอ ถูกต้องเสมอ
⚠️ ต้องเพิ่มข้อมูล Parent ก่อน Child เสมอ
💡

คำแนะนำการใช้ Referential Integrity

ควร เปิด Enforce Referential Integrity เสมอ เมื่อสร้าง Relationship ในระบบงานจริง เพราะช่วยป้องกันข้อผิดพลาดของข้อมูลได้อัตโนมัติ และเปิด Cascade Update ไว้ด้วยถ้าต้องการให้ PK เปลี่ยนได้ในอนาคต แต่ระวัง Cascade Delete เพราะอาจลบข้อมูลจำนวนมากโดยไม่ตั้งใจ

🎯

ผลการเรียนรู้: นักเรียนสามารถอธิบายความหมายของความสัมพันธ์ระหว่างตาราง จำแนกประเภท 1:1, 1:M, M:N กำหนด PK และ FK สร้าง Relationship ใน Access และเข้าใจการทำงานของ Referential Integrity ได้อย่างถูกต้อง

แบบรายงาน (Report) คือ การนำเสนอข้อมูลจากฐานข้อมูลในรูปแบบที่พร้อมพิมพ์หรือแสดงผล มักใช้สรุปข้อมูล สร้างใบเสร็จ หรือรายงานสถิติต่างๆ

ส่วนประกอบของ Report

Report Header ชื่อรายงาน, โลโก้ — แสดงครั้งเดียวที่หน้าแรก
Page Header หัวตาราง — แสดงทุกหน้า
Detail แถวข้อมูล — วนซ้ำสำหรับทุก Record
Page Footer หมายเลขหน้า — แสดงทุกหน้า
Report Footer สรุปรวม — แสดงครั้งเดียวท้ายรายงาน

⚡ แมโคร (Macro)

🤖

แมโครคืออะไร?

Macro คือชุดคำสั่งที่บันทึกไว้ล่วงหน้า เพื่อให้ทำงานซ้ำๆ โดยอัตโนมัติ ช่วยลดขั้นตอนการทำงาน เพิ่มประสิทธิภาพ

📋 Macro Actions ที่ใช้บ่อย

OpenForm
เปิด Form ที่กำหนด
OpenReport
เปิดหรือพิมพ์รายงาน
CloseWindow
ปิดหน้าต่างปัจจุบัน
GoToRecord
ไปยัง Record ที่กำหนด
RunQuery
รัน Query ที่กำหนด
MessageBox
แสดงข้อความแจ้งเตือน
FindRecord
ค้นหา Record ตามเงื่อนไข
Quit
ออกจากโปรแกรม Access

🛠️ วิธีสร้าง Macro

1

ไปที่แท็บ Create → Macro

2

เลือก Action จาก Dropdown ที่ต้องการ

3

กำหนด Arguments (พารามิเตอร์) ของ Action นั้นๆ

4

บันทึก Macro แล้วตั้งชื่อ เช่น mcrOpenStudentForm

5

นำ Macro ไปผูกกับปุ่มใน Form เพื่อใช้งาน
